There are two independent kinds of rate limit when you use Composio:

1. **Composio platform limits**: how many requests your organization can make to the Composio API per minute.
2. **Upstream provider limits**: limits the external service (Gmail, Slack, Google Ads, etc.) enforces on the calls Composio makes on your behalf.

## Composio platform limits

Rate limits are enforced **per organization** on a fixed one-minute window (aligned to the start of each minute), shared across all of your `userId`s, API keys, and endpoints. Your live limit is always returned in the `X-RateLimit` header; the table below lists the current defaults.

| Plan | Rate Limit | Window |
|------|------------|--------|
| Starter | 2,000 requests | 1 minute |
| Hobby | 2,000 requests | 1 minute |
| Growth | 10,000 requests | 1 minute |
| Enterprise | Custom | - |

<Callout>
All authenticated API endpoints share your organization's rate limit. This includes tool execution, connected accounts, triggers, and all other API operations.
</Callout>

### Rate limit headers

Every response includes headers, so you never have to hardcode a number:

| Header | Description |
|--------|-------------|
| `X-RateLimit` | Total requests allowed in the current window |
| `X-RateLimit-Remaining` | Requests remaining in the current window |
| `X-RateLimit-Window-Size` | Window size (e.g., `60s` for 60 seconds) |
| `Retry-After` | Seconds until the window resets (only on 429 responses) |

Treat these as the source of truth and drive your client off them.

### When you exceed the limit

You get an `HTTP 429 Too Many Requests` with a `Retry-After` header:

```json
{
  "message": "Rate limit exceeded. Limit: 10000 requests per 1 minutes"
}
```

A `429` is a throttle, not a cutoff. The window auto-resets the next minute and you can continue; nothing about your account changes.

### Will I get suspended for going over?

<Callout type="info">
**No.** Composio does not automatically suspend, review, or restrict accounts based on request volume, concurrency, or error rate. Exceeding your limit only returns `429` throttling that clears on the next minute. There is no automated abuse trigger.
</Callout>

The only hard block is a manual action by our team in rare cases (e.g. confirmed abuse), which returns `HTTP 403` with `Organization temporarily blocked. Please contact technical support.` It is never automatic, and support is the path to resolve it.

### Handling 429s

- Honor `Retry-After` first; otherwise wait for the window (`X-RateLimit-Window-Size`); otherwise fall back to exponential backoff with jitter, capping retries (e.g. 5) before surfacing an error.
- Watch `X-RateLimit-Remaining` and slow down before you hit zero.
- Cache tool definitions and other static data, batch where you can, and avoid tight polling loops.

```python
import time, random

def with_backoff(make_request, max_retries=5):
    for attempt in range(max_retries):
        resp = make_request()
        if resp.status_code != 429:
            return resp
        retry_after = resp.headers.get("Retry-After")
        delay = float(retry_after) if retry_after else min(2 ** attempt, 30) + random.random()
        time.sleep(delay)
    raise RuntimeError("Composio rate limit: retries exhausted")
```

## Upstream provider limits

These are separate from Composio's limits. When Composio calls Gmail, Slack, Google Ads, etc. on your behalf, that provider enforces its own limits and can return `429` / `RESOURCE_EXHAUSTED`, which Composio surfaces back to you.

The most common cause in production is **Composio's shared managed auth**: many organizations connect through the same shared OAuth app, so you share that app's rate-limit bucket (Slack in particular throttles non-Marketplace apps aggressively).

<Callout type="warn">
For production and high-volume workloads, use your own OAuth app via a [custom auth config](/docs/custom-app-vs-managed-app) instead of Composio-managed auth. Your own app gets its own isolated bucket and the provider's full per-app limits. Managed auth is best for development and prototyping.
</Callout>

## High-volume and multi-tenant usage

For multi-tenant apps, the recommended pattern is one Composio account with a unique `userId` per end-user. You do not need per-tenant sub-accounts. To scale throughput, move to a higher plan (which raises your per-org limit) and use your own OAuth apps for the providers you call heavily.
