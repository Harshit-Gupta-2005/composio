Composio stores an audit log of tool executions so you can observe and debug your agents. You control how much of each call is stored, including running with no payload retention at all.

## What Composio stores

When your agent runs a tool, Composio writes an execution log. Each log records:

- which toolkit and action ran, success or failure and the status, the connection and auth-config IDs, the `user_id` you supplied, timing, and the runtime/source;
- by default, the **request arguments** and the **response data** of the call.

Trigger events are logged similarly. Composio identifies your end users only by the opaque `user_id` you provide; it does not store end-user identities beyond that.

## How long we retain it

Tool execution logs and trigger event logs are retained for up to **one year**, after which they are automatically deleted.

## Log storage: control what we store

You control how much of each call is stored, per project, in **Settings → General → Log storage**:

- **Store all logs** (default) — the full request and response payloads are stored on the execution log.
- **Don't store data** — Composio does not persist the request arguments or the response data of your tool calls. It keeps only the audit record: which tool ran, when, success or failure, the IDs, and timing.

Selecting **Don't store data** lets you run Composio with **zero retention of tool-call payloads** while keeping an audit trail. The same setting governs [Proxy Execute](/docs/proxy-execute) logging, which fails closed: if the setting cannot be read, payloads are not stored.

<Callout type="info">
To enable: open the Dashboard, choose your project, go to **Settings → General → Log storage**, and select **Don't store data**.
</Callout>

## What "Don't store data" does and does not cover

- **Covered:** the request arguments and response data are dropped from the durable execution log.
- **Not covered** (this is inherent to running a tool call): while a call runs, the data is still sent to the destination provider and passes through Composio's execution path; file payloads may transit object storage; and trigger payloads are delivered to your configured destinations in real time. See [Sub-processors and data flow](/docs/security/sub-processors).

## Encryption

Connected-account credentials, auth configs, and API keys are encrypted at rest using AES-256-GCM, and all traffic is encrypted in transit using TLS.

## Stronger guarantees

For a contractual zero-data-retention agreement or shorter retention windows, contact your Composio account team or support. For our data-handling policies and the current sub-processor list, see the [Composio Trust Center](https://trust.composio.dev).
