Composio stores an audit log of tool executions so you can observe and debug your agents. You control how much of each call is stored, and you can choose not to store the payloads at all.

## Customer data vs. your end users' data

Throughout this guide, *you* are the Composio customer, and *your end users* are the people your agent acts on behalf of, each identified by the opaque `user_id` you supply. When a tool runs, its request and response often contain **your end users' data**. Composio never stores your end users' identities (only the `user_id` you provide), so their data lives entirely in the **tool-call payloads**, which is exactly what the **Log storage** setting below governs.

## What Composio stores

When your agent runs a tool, Composio writes an execution log. Each log records:

- which toolkit and action ran, success or failure and the status, the connection and auth-config IDs, the `user_id` you supplied, timing, and the runtime/source;
- by default, the **request arguments** and the **response data** of the call.

Trigger events are logged similarly.

## How long we retain it

Tool execution logs and trigger event logs are retained for up to **one year**, after which they are automatically deleted.

## Choose what we store: the Log storage setting

You control how much of each call is stored, per project, in **Settings → General → Log storage**:

- **Store all logs** (default) — Composio stores the full request and response payloads on the execution log.
- **Don't store data** — Composio does not store the request arguments or the response data of your tool calls. It keeps only the audit record: which tool ran, when, success or failure, the IDs, and timing.

With **Don't store data**, Composio keeps an audit trail but does not retain your tool-call payloads (including your end users' data) in its logs. The same setting governs [Proxy Execute](/docs/proxy-execute) logging, which fails closed: if the setting cannot be read, payloads are not stored.

![Log storage set to "Don't store data" in Project Settings, General](/images/log-storage-dont-store-data.png)

<Callout type="info">
To enable: open the Dashboard, choose your project, go to **Settings → General → Log storage**, and select **Don't store data**.
</Callout>

## Where your data goes

Choosing **Don't store data** stops Composio from persisting your payloads, but running a tool still moves data through a few places while the call executes:

- the **destination provider** you integrate with, which the tool call is made to;
- **Composio's execution path** while the call runs;
- **object storage** for file payloads, a **code sandbox** if you use the workbench, and your configured **trigger destinations**, in real time.

So "Don't store data" controls what Composio retains, not whether data is processed during execution. For the current list of our sub-processors and our data-handling terms, see the [Composio Trust Center](https://trust.composio.dev/subprocessors).
